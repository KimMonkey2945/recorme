# record MVP PRD

> ⚠️ **이 문서는 고정 명세가 아니라 현재 시점의 기준 스냅샷이다.** 범위·기능·데이터 모델·기술 스택은 개발 진행과 요구사항 변화에 따라 **얼마든지 변경될 수 있다.** 변경 시 `docs/`(api-contract·backend·database 등)와 함께 정합을 맞춘다.

## 핵심 정보

**목적**: 오늘 하루를 글로 남겨 두고 언제든 다시 꺼내볼 수 있는 개인 모바일 기록장. 형식에 얽매이지 않고 그날그날을 가볍게 기록한다.
**사용자**: 매일 짧은 글쓰기로 하루를 남겨 두고 싶은 모바일 사용자

## 사용자 여정

```
1. 앱 실행
   ↓ 로컬 토큰 유무 확인

   [토큰 없음 / 만료] → 로그인 페이지
   [토큰 유효 - 캐릭터 미선택] → 캐릭터 선택 페이지 (온보딩, go_router redirect 강제)
   [토큰 유효 - 캐릭터 선택됨] → 캐릭터 홈 (메인 탭 1번)

2. 로그인 페이지
   ↓ 소셜 로그인 버튼 탭 (카카오 / 구글)

   [최초 진입 - 미가입] → Supabase 세션 발급 → (백엔드 첫 요청 시 자동 가입) → 캐릭터 선택 페이지
   [재진입 - 기가입]    → Supabase 세션 발급 → 캐릭터 홈
   [인증 실패]          → 에러 토스트 후 로그인 페이지 유지

3. 캐릭터 선택 페이지 (온보딩 · 최초 1회)
   ↓ 캐러셀 (원숭이 / 레서판다 — 양옆 카드가 살짝 보이는 peek)
   ↓ 헤드라인 "기억을 같이 만들어갈 친구를 선택해주세요."

   [좌우 드래그 / 옆 카드 탭 / 하단 도트 탭] → 해당 캐릭터가 중앙으로(idle 애니메이션은 중앙 카드만)
   [선택]                  → 선택 저장(PUT /characters/me/selection) → 캐릭터 홈
   (둘 다 무료. 선택 전에는 다른 탭으로 나갈 수 없다)

4. 캐릭터 홈 (메인 탭 · 몰입형 "내 방")
   ↓ 사용자 행동

   [오늘 기록하기 CTA]  → 글 에디터 페이지 (오늘 날짜, 신규 작성 모드)
   [옷장 버튼]          → 옷장 페이지 (보유 아이템 착용·해제)
   [미션 버튼]          → 미션 페이지 (누적 업적 진행률·해금)
   [상점 버튼]          → 상점 페이지 (코인 구매. flag off면 비활성)
   [보상 알림 뱃지]      → 미확인 보상함 (코인·해금 아이템 확인)
   [회고 카드]          → 월간 회고 페이지 (이번 달 기록·감정·성장 요약)
   [하단 탭 이동]        → 캘린더 / 작심삼일 / 피드 / 프로필

5. 메인 캘린더 탭
   ↓ 날짜 탭 / 글쓰기 FAB / 하단 탭 이동

   [날짜 탭 - 확정 기록 있음]        → 글 상세 페이지
   [날짜 탭·FAB - 그 외]            → "글 작성 / 작심삼일 작성" 선택 시트
        ├ 글 작성       → 글 에디터 페이지 (신규 작성 모드)
        └ 작심삼일 작성  → 작심삼일 작성 페이지 (과거 날짜는 비활성)
   [하단 작심삼일 탭]                → 작심삼일 페이지 (진행/성공/실패 + 캘린더)
   (기록 있는 날짜엔 사용자가 입력한 감정 색 점(dot) 표시)

6. 글 에디터 페이지
   ↓ 사용자 행동

   [감정 선택 (선택 사항)] → 프리셋 칩 6종 택1 또는 "직접 입력"(≤20자) + 최근 사용 감정 추천
   [등록 버튼]            → DRAFT 저장(미확정·수정가능) → 이전 페이지
   [오늘을 기억하기 버튼]   → 확인 다이얼로그 → 확정(즉시 DONE·이후 수정 불가) → 상세 페이지
   [취소 버튼]            → 이전 페이지 (캐릭터 홈 / 메인 캘린더 / 글 상세 페이지)

7. 글 상세 페이지
   ↓ 사용자 행동

   [확정 직후]  → 캐릭터 리액션 오버레이 등장(대기 없음)
                  → 캐릭터 말풍선 대사(캐릭터별 상이) + 코인 적립 / 미션 해금 카드
                  → 탭하면 확인 처리(POST /rewards/ack) → 상세 본문
   [수정 버튼] → 글 에디터 페이지 (수정 모드, DRAFT만 가능)
   [삭제 버튼] → 삭제 확인 다이얼로그 → 확인 시 소프트 삭제 → 이전 페이지
   [뒤로가기]  → 이전 페이지 (메인 캘린더 페이지 또는 캐릭터 홈)

8. 프로필 탭
   ↓ 사용자 행동

   [프로필 수정]  → 닉네임·프로필 이미지·자기소개 편집 후 저장
   [로그아웃]     → 세션 삭제 → 로그인 페이지
```

## 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|--------------|------------|
| **F001** | 로그인 (소셜/이메일) | Supabase Auth로 소셜 로그인(카카오/구글) 또는 이메일 가입·로그인(확인 메일 필수), Supabase 세션 발급. 백엔드는 Supabase JWT 검증 + 최초 요청 시 자동 가입(이메일·소셜 동일 경로) (애플은 추후 확장) | 서비스 진입의 유일한 인증 관문 | 로그인 페이지 |
| **F002** | 캘린더 날짜 선택 | 월별 캘린더 표시, 기록 작성 여부를 날짜별로 시각 표시(**사용자가 입력한 감정 색 점(dot)** — 감정이 쓰이는 두 곳 중 하나), 날짜 탭으로 작성·조회 진입 | 날짜 기반 기록 작성의 핵심 진입점 | 메인 페이지 (② 캘린더 탭) |
| **F003** | 기록 작성 | 선택된 날짜에 자유 텍스트 기록 작성(본문 **최대 500자 하드 제한**). 해당 날짜 기록 미존재 시 신규 생성(INSERT). **[등록]**은 DRAFT(미확정·수정가능)로 저장, **[오늘을 기억하기]**는 확정(이후 수정 불가). 작성 중 **감정을 직접 입력**할 수 있다(F018, 선택 사항). **확정은 캐릭터 리액션·코인 적립·미션 판정의 유일한 트리거**(F028·F030·F031) | 서비스의 핵심 행위 | 글 에디터 페이지 |
| **F004** | 기록 목록 조회 | 작성된 기록을 날짜 역순으로 나열. 날짜와 내용 미리보기 표시, 커서 기반 무한 스크롤 | 과거 기록 탐색의 기본 수단 | 글 목록 페이지 |
| **F005** | 기록 단건 조회 | 특정 날짜 기록의 전체 내용 표시 | 작성한 글을 읽는 기본 기능 | 글 상세 페이지 |
| **F006** | 기록 수정 | 기존 기록 내용 수정(UPDATE). **DRAFT(미확정) 기록만 수정 가능**, 같은 날짜에 글 에디터 재진입 시에도 DRAFT면 수정으로 처리. **확정('오늘을 기억하기') 후에는 수정 불가**(삭제는 가능 → 같은 날짜 재작성) | 하루 1기록 원칙 구현, 오탈자·내용 변경 지원 | 글 에디터 페이지 |
| **F007** | 기록 삭제 | 기록 소프트 삭제(deleted_at 타임스탬프 기록). 확인 다이얼로그 후 처리. 삭제된 날짜는 다시 작성 가능. 첨부 사진 행·디스크 파일은 즉시 회수(복원 불가) | 잘못 작성된 기록 제거·디스크 공간 절약 | 글 상세 페이지 |
| **F012** | 기록 사진 첨부 (본문 인라인) | 기록 본문에 사진을 **인라인 임베드**(작성 중 `POST /diaries/images`로 즉시 업로드 → 반환 `{url}`을 Delta에 삽입, jpg/png/webp). 바이너리는 스토리지에 저장하고 DB엔 경로만 보관하며 본문 `content`(Delta)가 이미지의 **단일 진실 공급원**. 기록 삭제·수정 시 본문에서 빠진 이미지의 디스크 파일 회수(별도 `diary_images` 1:N 테이블은 폐지) | 글과 함께 그날의 사진 기록 | 글 에디터/상세 페이지 |
| **F011** | 프로필 조회·수정 | 내 프로필(닉네임·프로필 이미지·자기소개 bio) 조회 및 수정. `GET/PUT /users/me`. 가입 시 JIT로 생성된 정보를 사용자가 직접 갱신 | 가입 정보의 사용자 관리·표시 이름 변경 | 프로필 페이지 |

### 3. 작심삼일(3일 결심) 기능

> 기록(diary)과 **독립된 부가 기능**. 날짜 판정은 전부 **서버 KST(Asia/Seoul) 권위**이며, 완료 체크는 날짜 인자 없이 서버 '오늘'로 판정한다.

| ID | 기능명 | 설명 | 관련 페이지 |
|----|--------|------|------------|
| **F013** | 작심삼일 생성·수정 | **시작일(오늘/미래, 과거 불가) + 할일 제목 + 알림 시각(선택)**으로 3일 결심 생성. 생성 시 3일치 체크(day 1~3, `PENDING`)를 프리생성. 진입은 캘린더 날짜 탭·글쓰기 FAB의 **"글 작성 / 작심삼일 작성" 선택 시트**에서 분기. **진행 중(ONGOING) 결심은 제목·알림 시각만 수정 가능**(`PUT /resolutions/{id}`) — 시작일 변경은 종료일·체크 재계산 복잡도로 미지원(삭제 후 재작성으로 유도) | 작심삼일 작성/수정 페이지 |
| **F014** | 완료 체크 · 성공/실패 판정 | 진행 중 결심의 **오늘 체크를 '완료'(멱등)**. 3일 모두 완료 시 즉시 `SUCCESS`, 하루라도 그 날(KST 자정) 안에 못 누르면 자정 배치가 `MISSED`+결심 `FAILED` 처리 | 작심삼일 상세 페이지 |
| **F015** | 연장 (streak) | **성공한 결심만** 같은 할일로 '다음 3일'을 이어가 새 작심삼일 생성(시작일 `max(이전 종료+1, 오늘)`). `streak_seq`로 연속 N회 표시. 이중 연장 차단 | 작심삼일 상세 페이지 |
| **F016** | 작심삼일 목록·캘린더 조회 | **진행중/성공/실패** 세그먼트 리스트(3일 진행 도트·streak 배지, 커서 페이징) + **월별 캘린더**로 결심을 선택·조회 | 작심삼일 탭 |
| **F017** | 리마인더·완주 푸시 (FCM) | **미완료 시 매일 리마인더**(알림 시각 도래, 하루 1회 멱등), **3일 완주 축하**, 실패 안내를 **FCM 서버 푸시**로 발송. 폰 상단 알림 탭 시 해당 결심 상세로 딥링크 | (시스템 알림 → 작심삼일 상세 페이지) |

### 3.5 감정 (사용자 직접 입력) — ~~감정 분석 · 동적 테마 (LLM)~~

> ⚠️ **Phase 7에서 축소됨.** 기록 확정 시 멀티모달 LLM이 감정을 분석해 테마·연출을 자동으로 입히던 구조를 **비활성화**하고, **감정은 사용자가 직접 입력**하는 순수 기록 메타데이터로 바꿨다. 연출의 주인공은 감정이 아니라 **내 캐릭터**(§3.6)로 일원화한다.
>
> **번복 이력**: 초기 PRD "MVP 이후 제외" → Phase 5에서 **구현 완료**되어 승격(F018/F019) → **Phase 7에서 비활성화/대체**.
> LLM 코드·`emotion_types` 마스터·`diaries`의 감정/테마 컬럼은 **삭제하지 않고 보존**하며, `record.analysis.enabled=false` 플래그로 차단만 한다(`ANALYSIS_ENABLED=true` 한 줄로 복구 가능).
> 감정은 **캐릭터 리액션·미션 판정·해금 어디에도 쓰이지 않는다**. 오직 **캘린더 점 색**과 **월간 회고 통계**에만 쓰인다.

| ID | 기능명 | 설명 | 관련 페이지 |
|----|--------|------|------------|
| **F018** | ~~감정 분석 (멀티모달 LLM)~~ → **감정 직접 입력** | ~~확정 기록의 순수 텍스트 + 본문 인라인 이미지를 LLM에 입력해 대표 감정·테마색·AI 코멘트/제목·무드 이모지·감정 점수 분포를 생성~~ → **Phase 7에서 `record.analysis.enabled=false`로 비활성화**(코드·`emotion_types` 테이블·`diaries` 감정 컬럼 보존). 대신 **사용자가 작성 중 감정을 직접 입력** — **프리셋 6종(기쁨/슬픔/분노/평온/불안/중립) 택1** 또는 **자유 텍스트 직접 입력(≤20자, `diaries.emotion_label`)** + 최근 사용 감정 추천. **둘 다 선택 사항**(감정 없이도 확정 가능), 동시 지정은 400 `EMOTION_CONFLICT`. flag off 시 확정하면 **즉시 `DONE`**(분석 대기 없음), AI 필드(테마색·코멘트·제목·이모지·점수)는 NULL | 글 에디터 페이지 |
| **F019** | ~~감정 기반 동적 테마 · 연출~~ → **캐릭터 리액션(F031)으로 대체** | ~~감정 배경/글자/강조 색 동적 적용, 감정 마스코트 영상, 상세 시네마틱 인트로, 분석 대기 러닝 로딩 영상, PENDING 3초 폴링~~ → **Phase 7에서 전부 제거**(셰이더·감정 mp4 6종·동적 테마 팔레트·인트로·러닝 영상 폐기). 연출은 **내 캐릭터 하나로 일원화**되어 **F031 기록 리액션**이 대체한다. 감정은 시각적으로 **캘린더 점 색**·**상세/피드 카드의 감정 칩**·**월간 회고 통계**에만 남는다 | 캐릭터 홈·글 상세·메인 페이지 |

### 3.6 캐릭터 (커스터마이징 · 미션 해금 · 락인)

> **Phase 7의 중심.** "기록하면 LLM이 감정을 분석해 테마를 입힌다"에서 **"기록하면 내 캐릭터가 반응하고, 쌓일수록 캐릭터가 꾸며진다"**로 서비스의 축을 옮긴다. 애착 기반 7일 리텐션과, 데이터가 쌓일수록 떠나기 어려운 락인 구조가 목표다.
>
> **기록 확정('오늘을 기억하기')이 모든 것의 트리거**다 — 리액션·코인 적립·미션 판정·해금이 확정 1회에서 파생된다(확정 후 수정 불가 규칙을 유지해 보상 어뷰징을 막는다). 캐릭터는 **Rive 비트맵 리깅**으로 렌더하며, 에셋 미비 시 `PlaceholderCharacterView`(PNG 합성)로 폴백한다.

**캐릭터 2종 (Phase 7 시작 기준)**

| 코드 | 이름 | 성격 | 말투 |
|------|------|------|------|
| `MONKEY` | 원숭이 | **여유롭고 느긋한** 성격. 하루를 놓쳐도 다그치지 않고 "천천히 해도 된다"고 말한다 | 느긋하고 능청스러운 반말 |
| `RED_PANDA` | 레서판다 | **부지런하고 애착이 강한** 성격. 사용자의 연속 기록을 자기 일처럼 챙기고 기다린다 | 애쓰고 살뜰하게 챙기는 말투 |

> **캐릭터별로 리액션 대사가 다르다.** 같은 상황(확정/연속 기록/미션 달성/레벨업)에서도 원숭이와 레서판다가 서로 다른 대사를 낸다(`character_lines.character_code`). 대사 선택은 **감정이 아니라 맥락**(`CONFIRM` / `STREAK_3` / `STREAK_7` / `MISSION` / `LEVEL_UP` / `IDLE`) 기반이다. 성격 대비가 캐릭터 선택의 유일한 실질적 차이이므로, 대사 톤은 두 캐릭터를 명확히 갈라 작성한다.
>
> **2종으로 시작하는 이유**: 두 캐릭터는 체형이 다르다(레서판다가 통통하고 팔이 짧다). 따라서 **옷 1종에 캐릭터별 PNG가 각각 필요**하다 — 캐릭터가 늘면 의상 에셋이 곱셈으로 늘어난다. 캐릭터 추가는 아이템 수가 적을 때 신중히 판단한다.

| ID | 기능명 | 설명 | 관련 페이지 |
|----|--------|------|------------|
| **F026** | 캐릭터 선택 | **원숭이 / 레서판다 2종** 중 1종 선택(둘 다 무료). **회원가입 직후 온보딩**에서 좌우 2장 대형 비교로 고른다. `selectedCharacter == null`이면 라우터가 온보딩으로 강제 리디렉션. 이후에도 캐릭터 교체 가능하며, **교체해도 보유 아이템·착용 상태는 유지**된다(소유는 `group_code` 단위, 렌더만 캐릭터별 variant로 재해석) | 캐릭터 선택 페이지, 캐릭터 홈 |
| **F027** | 코스튬 · 옷장 | 보유 아이템을 슬롯별로 착용·해제·교체. 슬롯: **`HAT`(모자) / `OUTFIT`(의상) / `GLASSES`(안경) / `PROP`(손 소품) / `ROOM_PROP`(방 소품, 0~5개 다중 진열) / `BACKGROUND`(배경)**. 단일 슬롯은 1개만, `ROOM_PROP`만 다중. 착용은 **`group_code` 단위**이고, 렌더 시 `(group_code + 내 캐릭터)` 조합으로 variant 이미지를 고른다. 내 캐릭터용 variant가 없으면 409 `ITEM_VARIANT_MISSING` | 옷장 페이지, 캐릭터 홈 |
| **F028** | 코인 재화 | 앱 내 단일 재화. **기록 확정**(기본 10코인)과 **작심삼일 완주**(기본 30코인)로 적립. 잔액은 `user_wallets`, 증감 이력은 `character_events`가 원장(ledger)으로 겸한다. **적립은 항상 동작**하고 소비(상점)만 플래그로 게이팅한다. `(user_id, event_key)` 유니크 게이트로 **중복 적립 불가**(재전달·폴러 보정에도 멱등) | 캐릭터 홈(코인 표시), 상점 페이지 |
| **F029** | 상점 | 코인으로 아이템 group을 구매(`acquire_type=COIN`). 잔액 부족 시 409 `COIN_INSUFFICIENT`(경합 시에도 음수 잔액 불가). **초기에는 `record.character.coin-enabled=false`로 비활성**(구매 시 403 `FEATURE_DISABLED`) — 코인은 먼저 쌓이게 두고, 해금 밸런스가 잡힌 뒤 상점을 연다 | 상점 페이지 |
| **F030** | 미션(누적 업적) 해금 | **아이템 해금의 유일한 경로.** 누적 업적 달성 시 코인 또는 아이템 group을 지급한다. 판정 규칙은 `missions.rule`(JSONB)에 선언하고 `MissionEvaluator` 순수 함수가 평가 — `DIARY_COUNT`(확정 기록 누적) / `CONSECUTIVE_DAYS`(연속 기록일) / `RESOLUTION_SUCCESS`(작심삼일 완주 횟수) / `RESOLUTION_STREAK`(연장 연속 횟수) / `LEVEL`. **감정 기반 규칙은 없다.** 진행률은 `user_progress` 캐시로 O(1) 판정. 달성 보상은 1회만(`event_key='MISSION:{code}'` 게이트) | 미션 페이지, 캐릭터 홈 |
| **F031** | 기록 리액션 | **기록 확정 즉시** 캐릭터가 등장해 **말풍선으로 반응**한다(F019 감정 연출의 대체재). 확정 응답이 곧 `DONE`이므로 **분석 대기·폴링 없이 즉시** 재생. 오버레이에 캐릭터 애니메이션(`react`/`celebrate` 트리거) + **캐릭터별·맥락별 대사 1줄** + 적립 코인 / 해금 미션 카드를 함께 띄우고, 탭하면 확인 처리(`POST /rewards/ack`). **획득이 없어도 대사 1줄은 항상 나온다**(리액션은 보상이 아니라 애착의 장치) | 글 상세 페이지 |
| **F032** | 월간 회고 · 성장 | 이번 달 **기록 수·연속 기록일·감정 분포(사용자가 입력한 감정)·작심삼일 완주·획득 아이템·레벨 성장**을 한 장으로 요약. 캐릭터가 그 달을 함께 돌아보는 형식. **락인 장치** — 쌓인 기록이 많을수록 잃을 것이 커진다. 레벨/경험치는 기록 확정으로 누적(`user_character_state.level/exp`, 기본 확정당 10exp) | 월간 회고 페이지, 캐릭터 홈 |
| **F033** | 캐릭터 홈 · 소품 진열 | **앱의 메인 화면(1번 탭)**. 몰입형 풀스크린 "내 방" — 착용 `BACKGROUND` 위에 중앙 대형 캐릭터(idle: 숨쉬기·눈 깜빡임·두리번거림), 주변에 `ROOM_PROP` 소품 진열, 상단 반투명 상태바(레벨·성장 게이지·코인·미확인 보상 뱃지), 하단 플로팅 패널(**"오늘 기록하기" 주 CTA** + 옷장·미션·상점 진입) | 캐릭터 홈 |

### 캐릭터 도메인 (Phase 7 신설 — V16~V18)

> **핵심 설계 포인트 ①: 아이템은 `group_code`(소유·착용) ↔ `variant`(렌더) 2단 구조다.**
> 원숭이와 레서판다는 **체형이 다르다**(레서판다가 통통하고 팔이 짧다). 그래서 같은 "빨간 후드티"라도 **캐릭터마다 옷 PNG를 따로 그려야 한다.** 그렇다고 사용자에게 "원숭이용 후드티"와 "레서판다용 후드티"를 별개 아이템으로 보여줄 수는 없다 — 캐릭터를 바꾸면 옷장이 통째로 비어버리기 때문이다.
> → **소유·착용·구매·해금은 전부 `item_groups.code`("빨간 후드티") 단위**로 다루고, **화면에 그릴 때만 `(group_code + 내 캐릭터)` 조합으로 `character_items`에서 variant 이미지를 골라 렌더**한다. 사용자에겐 **group 하나로만 보인다.** 캐릭터를 교체해도 옷장과 착용 상태는 그대로 따라오고, 이미지만 새 체형에 맞게 재해석된다. 내 캐릭터용 variant가 아직 제작되지 않았다면 409 `ITEM_VARIANT_MISSING`으로 명시적으로 실패시킨다.
> **캐릭터를 추가하면 기존 모든 옷의 variant를 새로 그려야 한다** — 이것이 캐릭터를 2종으로 시작하는 이유다.
>
> **핵심 설계 포인트 ②: `character_events`가 모든 부작용의 단일 관문이다.**
> `(user_id, event_key)` 유니크 제약 한 줄이 ① **멱등 게이트**(중복 적립 원천 차단) ② **코인 원장** ③ **리액션 페이로드** ④ **미확인 보상 알림함**을 겸한다. 코인 적립·미션 해금·상점 구매가 전부 이 게이트를 통과해야만 실행된다.
>
> **핵심 설계 포인트 ③: 감정은 캐릭터 도메인에 존재하지 않는다.** 대사 선택(`character_lines.context`)도 미션 판정(`missions.rule`)도 **맥락과 누적 수치**만 본다.

| rule | 판정 규칙 JSONB — `{"type":"DIARY_COUNT","threshold":10}`. type: DIARY_COUNT / CONSECUTIVE_DAYS / RESOLUTION_SUCCESS / RESOLUTION_STREAK / LEVEL. **감정 규칙 없음**. `MissionEvaluator` 순수 함수가 평가(DB 트리거 없음) | JSONB |
| coin_reward | 지급 코인 | INT |
| item_group_reward | 해금 아이템 그룹 (**variant가 아니라 group**) | → item_groups.code (nullable) |
| sort_order / active | 정렬·노출. `CHECK(coin_reward > 0 OR item_group_reward IS NOT NULL)` — 보상 없는 미션 금지 | INT / BOOLEAN |

| user_missions | 설명 | 타입/관계 |
|------|------|----------|
| user_id + mission_code | 달성 기록 PK (복합) | → users.id, → missions.code |
| achieved_at | 달성 시각 | TIMESTAMP |

#### user_character_state / user_item_groups / user_equipment (내 캐릭터)

| user_character_state | 설명 | 타입/관계 |
|------|------|----------|
| user_id | PK | → users.id |
| selected_character | 현재 선택한 캐릭터 (NULL이면 온보딩 미완료 → 라우터가 캐릭터 선택으로 강제) | → characters.code |
| level / exp | 레벨·경험치 (기록 확정당 기본 10exp) | INT |

| user_item_groups | 설명 | 타입/관계 |
|------|------|----------|
| user_id + group_code | **보유는 group 단위**(복합 PK) — 캐릭터를 바꿔도 옷장이 유지되는 근거 | → users.id, → item_groups.code |

| user_equipment | 설명 | 타입/관계 |
|------|------|----------|
| user_id + slot + slot_index | 착용 상태 (복합 PK). `CHECK(slot='ROOM_PROP' OR slot_index=0)` — **단일 슬롯은 1개, ROOM_PROP만 0~5 다중 진열** | → users.id |
| group_code | **착용도 group 단위.** 렌더 시 (group_code + selected_character)로 variant 해석 | → item_groups.code |

#### user_progress / user_wallets (미션 판정 캐시 · 지갑)

| user_progress | 설명 | 타입/관계 |
|------|------|----------|
| user_id | PK | → users.id |
| confirmed_diary_count | 확정 기록 누적 수 (DIARY_COUNT 판정) | INT |
| consecutive_days / last_confirmed_date | 연속 기록일 + 마지막 확정 날짜 (CONSECUTIVE_DAYS 판정. 하루 건너뛰면 리셋) | INT / DATE |
| resolution_success_count / max_streak_seq | 작심삼일 완주 횟수·최대 연장 순번 (RESOLUTION_* 판정) | INT / SMALLINT |

> 미션 판정을 **O(1)**로 만들기 위한 비정규화 캐시. 매 확정마다 집계 쿼리를 돌리지 않는다.

| user_wallets | 설명 | 타입/관계 |
|------|------|----------|
| user_id | PK | → users.id |
| balance | 코인 잔액. `CHECK(balance >= 0)` — 경합 상황의 최종 방어선(소비는 `UPDATE ... WHERE balance >= ?`로 0행이면 `COIN_INSUFFICIENT`) | INT |

#### character_events (멱등 게이트 · 코인 원장 · 리액션 소스 · 보상 알림함)

| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| id | 내부 PK | BIGINT |
| user_id | 소유자 참조 | → users.id |
| event_key | **멱등 키** (`DIARY_CONFIRM:{diaryId}`, `MISSION:{code}`, `RESOL_SUCCESS:{id}`, `PURCHASE:{groupCode}:{n}`). **`UNIQUE(user_id, event_key)`가 이 도메인 전체의 안전장치** — 삽입 성공(1행)만이 모든 부작용의 진입 조건 | TEXT |
| event_type | 이벤트 종류 (DIARY_CONFIRM / RESOLUTION_SUCCESS / MISSION / PURCHASE / LEVEL_UP) | VARCHAR |
| coin_delta / balance_after | 코인 증감과 처리 후 잔액 — **이 테이블이 곧 코인 원장(ledger)** | INT |
| diary_id | 확정 이벤트의 기록 참조 (리액션 조회 키) | → diaries.id (nullable) |
| payload | 리액션 페이로드 — **캐릭터 대사 1줄 + 달성 미션 + 코인 결과**. 앱 리액션 오버레이의 단일 소스 | JSONB |
| acked_at | 사용자가 보상을 확인한 시각 (NULL = 미확인 → 캐릭터 홈 알림 뱃지) | TIMESTAMP |

> 캐릭터 도메인은 기록(diary)·작심삼일(resolution)을 **모른다**(단방향). 기존 서비스는 `DiaryConfirmedEvent` / `ResolutionSucceededEvent`를 발행만 하고, 캐릭터 리스너가 **AFTER_COMMIT + @Async**로 수신한다. 따라서 **보상 처리가 실패해도 기록 저장은 롤백되지 않는다.** 유실분은 백스톱 폴러가 보정하며, 재전달되어도 위 유니크 게이트가 중복 적립을 막는다.

  - **폴백**: 에셋/리깅 미완 구간과 **웹(`kIsWeb`, `rive_native` wasm 제약)**에서는 `PlaceholderCharacterView`(PNG Stack 합성 + `render_meta` 좌표)로 전 기능을 개발·테스트한다. `--dart-define=USE_RIVE=false`가 기본값
